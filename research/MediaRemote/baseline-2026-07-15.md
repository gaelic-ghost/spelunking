## Framework Binary Layout

The active system framework directories are not ordinary binary-bearing framework directories on this machine:

- `MediaRemote.framework/MediaRemote` is a symlink to `Versions/Current/MediaRemote`, but `Versions/A/MediaRemote` is absent.
- `MediaRemoteDaemonServices.framework/MediaRemoteDaemonServices` is a symlink to `Versions/Current/MediaRemoteDaemonServices`, but `Versions/A/MediaRemoteDaemonServices` is absent.
- The dyld shared-cache map contains entries for both framework install names:
  - `/System/Library/PrivateFrameworks/MediaRemote.framework/Versions/A/MediaRemote`
  - `/System/Library/PrivateFrameworks/MediaRemoteDaemonServices.framework/Versions/A/MediaRemoteDaemonServices`

Inference: live binary inspection should use dyld shared-cache tooling or extracted cache images, not the top-level framework symlink.

## Support Binary Entitlements

`mediaremoted` is heavily privileged. Particularly relevant entitlements and allowances include:

- `com.apple.mediaremote.group-sessions`
- `com.apple.mediaremote.nearby-device`
- `com.apple.mediaremote.remote-control-discovery`
- `com.apple.mediaremote.send-commands`
- `com.apple.mediaremote.set-now-playing-app`
- `com.apple.mediaremote.set-playback-state`
- `com.apple.mediaremote.ui-control`
- `com.apple.mediaremote.ui-server-connection`
- `com.apple.avfoundation.allow-system-wide-context`
- `com.apple.avfoundation.allows-access-to-device-list`
- `com.apple.avfoundation.allows-set-output-device`
- `com.apple.airplay.receiver.mediaremote.services`
- mach lookup access for `com.apple.mediaremoted.xpc`, `com.apple.mediaremoteui.services`, multiple CoreMedia endpoint/routing services, AirPlay services, IDS, SessionServices, and related system services.

`mediaremoteagent` did not print embedded entitlements with the same `codesign` command.

Practical boundary: daemon entitlements show what Apple grants to its own service, not what an ordinary userland helper can do. First client experiments should remain read-only.
